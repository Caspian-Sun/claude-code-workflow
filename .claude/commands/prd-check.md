你现在是 PRD 完备性检查器。对输入的 PRD 文件执行硬性闸门检查, 输出是否可以进入 `/plan` 任务拆解阶段。

## 适用场景

1. **用户审阅 PRD 过程中实时自检** — 改一点跑一下, 知道离「能跑 `/plan`」还差多少
2. **`/plan` 命令的前置校验** — `/plan` 执行前必须先跑本命令, 不通过直接拒绝进入任务拆解

## 输入

- `@docs/prds/xxx.md` 路径 → 直接读取
- 不带路径 → 停下询问: 「请指定要检查的 PRD 文件路径, 例: /prd-check @docs/prds/login.md」

## 检查项 (按顺序执行, 不短路, 一次报全部问题)

依次执行以下 5 项检查, 每项都跑完, 最后汇总输出。**不要遇到第一个失败就停**, 用户希望一次看到所有问题。

### 检查 1: `[待确认]` 零命中 (P0, 必须通过)

全文 grep `[待确认]`, 命中数必须为 0。

- **为什么严**: `[待确认]` 会一路污染到 `task.businessRules` → 源码 `@rules` → 测试 `it()` 断言, 下游全错。入口拦截成本最低。
- **不通过时**: 列出每个命中的行号和原文片段

### 检查 2: `[待填写]` 正文零命中 (P1)

全文 grep `[待填写]`, **允许**出现在:
- 元信息表 (「负责人」字段)
- 变更记录表 (「变更人」列)

**不允许**出现在其他任何正文位置。

- **为什么允许元信息**: 这些字段是协作信息, 不影响下游生成
- **不通过时**: 列出越界的命中位置

### 检查 3: 业务规则章节不含占位符 (P0)

扫描所有 `### 业务规则` 章节下的条目, 每条不得包含:
- `[待确认]`
- `TODO` / `FIXME` / `???`
- 空条目 (如 `1. `)

- **为什么严**: `businessRules` 是任务拆解和测试生成的直接来源, 带占位符等于下游断言不可靠
- **不通过时**: 定位到哪个功能点的哪条规则

### 检查 4: 数据契约状态列齐全 (P1)

扫描所有「调用的接口」表格, 每个 operationId 必须标状态, 取值只能是:
- `✅ 已存在`
- `🆕 待后端实现`

- **不通过时**: 列出无状态或状态取值非法的 operationId

### 检查 5: 🆕 接口有 stub (P1)

对每个 `🆕 待后端实现` 的 operationId, 检查它是否:
- 出现在 PRD 的「接口提议」章节 (代码块里有对应 operationId)
- **或**已在 `workspace/api-spec/openapi.local.json`

两处都没有 → 不通过。

- **不通过时**: 列出缺 stub 的 operationId

### 检查 6: 上游文档忠实性 (P0, 🔴 必须通过)

> 依据 [.claude/rules/upstream-fidelity.md](../rules/upstream-fidelity.md), 防 PRD 自由联想 / 覆盖产品文档原意。

#### 6.A 引用真实性 (P0)

扫描 PRD 中所有指向上游的引用:
- `产品文档 §...` / `产品文档 §「...」`
- `设计稿 ...png` / `docs/designs/...`
- `OpenAPI ...json`

对每条引用:
- ☐ 引用的文件**真实存在** (Read 检查)
- ☐ 引用的章节 / 锚点**在文件中真实存在** (grep 检查)
- ☐ 优先要求引用带行号 (例: `wallet-spec.md:367`); 仅 `§「X」` 形式必须能 grep 到匹配章节

**不通过示例**:
```
home.md L167: 产品文档 §「钱包切换入口」需求
  → grep "钱包切换入口" docs/prds/_imports/  →  0 命中
  → 🔴 引用的章节不存在
```

#### 6.B 虚假评审拦截 (P0)

扫描 PRD 全文以下短语, 命中后**必须**有对应评审证据:
- 「与产品确认」/ 「与设计确认」/ 「与产品沟通」
- 「产品已确认」/ 「产品决定」/ 「产品拍板」
- 「评审通过」/ 「评审已确认」
- 「与 X 同学确认」 (X 为人名)

**评审证据**只接受以下之一:
- 同一文件「## 变更记录」表里有对应行 (日期 / 评审人 / 决策)
- 关联 commit message 含 `Reviewed-by:` 或评审人 trailers
- PR/MR 上有产品评审人 approval

#### 6.C 业界惯例拦截 (P1)

扫描 PRD 中是否把「业界做法」当作已定规则:
- 「业界主流是 X」/「参考 MetaMask / Trust Wallet / 微信 / 支付宝 ...」
- 「类似 X 的做法」/「市面上一般这么做」
- 「这样更好 / 更顺手 / 更省步数」

这类表述**只能出现在 `## 待评审建议` 节**, 不能作为业务规则正文写。

#### 6.D 冲突显式化 (P1)

扫描 PRD 是否含 `## 冲突待决` 节:
- 若有 → 阻塞 `/plan` (硬阻塞), 必须先解决
- 若无 → 检查 PRD 与产品文档对应字段是否一致 (针对关键交互):
  - 跳转路径 (点 X → 跳 Y)
  - 字段名 / 接口名
  - 流程步骤
  - 命中差异时 → 提示「未列入 `## 冲突待决`, 可能未察觉」

## 附加检查 (软提示, 不阻塞)

以下问题只警告, 不影响 pass/fail:

- `[默认假设]` 命中数 (提醒评审会需要确认)
- 元信息「负责人」是否 `[待填写]`
- 变更记录「变更人」是否 `[待填写]`

```

  ✅ 业务规则无占位符
  ✅ 数据契约状态列齐全 (5 个 operationId)
  ✅ 🆕 接口 stub 完整 (5 个)
  ✅ 上游文档忠实性 (引用真实 / 无虚假评审 / 无业界惯例混入 / 无未列冲突)

  • 文中有 6 处 [默认假设], 评审会请逐条确认
  • 元信息「负责人」仍为 [待填写]

```

## 设计原则

- **不修改 PRD 文件**, 只读检查
- **不短路**: 一次跑完 5 项, 全部问题一起报
- **输出 actionable**: 每条问题带行号/位置 + 指向 `docs/prds/REVIEW.md`
- **默认假设不拦**: AI 已经给了默认值, 可以带着走, 评审会再定
