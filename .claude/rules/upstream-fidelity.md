# 上游文档忠实性 (Upstream Fidelity)

> **核心原则**: 上游文档 (产品文档 / 设计稿 / OpenAPI) 是 **单一事实源 (single source of truth)**。
> 下游产出 (PRD / 任务清单 / 代码) **必须严格遵循上游意图, 不允许自行联想 / 重构 / 覆盖**。
> 当发现冲突 / 模糊 / 缺漏时, **必须停下来给用户选项, 由用户决定**, 不允许 AI 自行拍板。

---

## 为什么有这条规则

历史教训反复证明: 起草 PRD 的人 (无论是 AI 还是人) 在面对上游文档不够明确的地方时, 容易出现以下行为:

- **覆盖原意**: 把上游「跳 A」自己改成「跳 B」, 觉得 B 更合理
- **虚构依据**: 引用上游不存在的章节, 给自己的改动套外壳
- **造伪评审**: 写「与产品确认」「评审通过」但实际没有任何评审证据
- **跨段重构**: 把上游「内部子入口 X」改写成「主入口 Y」, 改变层级
- **业界惯例当规则**: 「MetaMask 这么做, 所以我们也这么做」直接成 PRD 规则

这些行为表面看是「优化」, 实质是**让产品意图在 PRD → plan → code 链路上无声丢失**, 最终用户拿到的产品跟原产品文档意图不一致, 而且**因为 PRD 看起来"完整 + 有引用"反而难以发现是错的**。

---

## 适用范围 (3 类上游文档)

| 上游文档 | 下游产出 | 适用命令 |
|---------|---------|---------|
| **产品文档** (产品同学写的 spec / Notion / 飞书等导入到 `docs/prds/_imports/`) | PRD (`docs/prds/**/*.md`) | `/prd`, `/prd-check` |
| **设计稿** (Figma / 截图等放在 `docs/designs/`) | PRD + 任务 + 代码 | `/prd`, `/plan`, `/code`, `/review` |
| **OpenAPI 协议** (`docs/apis/openapi/*.json`) | API md + Repository / DTO | `/code`, `/review` |

## 硬规则 (Hard Rules)

### R1. 上游原文必须可追溯, 引用必须真实

- 下游产出引用上游时, **必须给出原文位置** (文件路径 + 行号或锚点)
- 引用的章节 / 段落 **必须实际存在**, 不允许引用虚构章节
- 引用前 grep / 看一眼 实际文件, 确认存在

❌ 反例:
```
点击钱包名 → 跳「钱包切换页」(产品文档 §「钱包切换入口」需求)
```
("§「钱包切换入口」" 章节在原产品文档里根本不存在)

✅ 正例:
```
点击钱包名 → 跳「钱包切换页」(产品文档 wallet-spec.md:367)
```
(引用真实行号, 可一键跳到原文核对)

### R2. 不允许「自由联想 / 重构上下文」

下游不允许:
- 把上游 A 段的话 **改写 / 重命名** 套到 B 段去
- 把上游「内部子入口 X」**重构成「主入口 Y」**
- 自由发挥添加上游没说的字段 / 流程 / 跳转

❌ 反例: 产品文档「**个人中心-设置页 →** Switch Wallet」被重构成「**首页顶部 →** 钱包切换页」, 把内部入口当成主入口

### R3. 「与产品确认」必须有评审证据

下游产出**禁止凭空写**:
- 「与产品确认」
- 「与设计确认」
- 「产品决定 X」
- 「评审通过」

除非有以下之一作证:
- commit message 里有评审人 (例: `Reviewed-by: @产品xx`)
- PRD 末尾「变更记录」表有一行: 日期 / 评审人 / 决策内容
- 关联的 GitLab MR / GitHub PR 里有产品评审人 comment / approval

无证据时, 措辞应改成:
- 「**PRD 作者建议** X, 待产品评审」
- 「**与产品文档冲突**, 待用户确认 (见 §冲突待决)」

### R4. 上游与设计稿冲突时, 必须显式标注 + 停下问

如果上游产品文档 ≠ 设计稿 (例: 产品文档说跳 A, 设计稿暗示跳 B), **不允许 AI 自行选一边**, 必须:
1. 在 PRD 加 `## 冲突待决` 节, 明确列冲突点
2. `/prd-check` 见此节自动阻塞进入 `/plan`
3. 等用户回答 (在 PRD 变更记录加一行)

### R5. 「业界惯例」「我觉得」「这样更好」**不是依据**

下游产出禁止以「业界 X 这么做」「这样更顺手」「减少步数」等理由覆盖上游意图。可以作为**提议**写在「待评审」段落里, 但**不能作为已确定的 PRD 规则**。

---

## 与设计稿冲突时的特殊处理

按 `.claude/commands/prd.md` 第一步 A 「设计稿元素清单」与设计稿真意有冲突时:
- 设计稿元素清单本身 ≠ 设计稿真意, 只是 AI 从图中识别出来的清单
- 若识别与产品文档冲突, 优先信**产品文档原文**, 在 PRD 加「设计稿与产品文档冲突待决」节
- 等用户决定后再展开

## 检查清单 (`/prd-check` + `/plan-check` 应跑)

### A. 引用真实性 (P0)

- ☐ PRD 里所有「产品文档 §X」/「设计稿 X」/「H5 X」引用, 章节 / 文件 **真实存在**
- ☐ 引用带行号或锚点, 不带模糊「§某段」表述

### B. 自由发挥拦截 (P0)

- ☐ grep 「与产品确认」「与设计确认」「产品决定」「评审通过」, 命中后**必须**有对应评审证据 (commit / 变更记录 / MR)
- ☐ grep 「自行改」「重构为」「调整为」「业界主流」「参考 X」, 命中后必须明确标「待评审」

### C. 冲突显式化 (P1)

- ☐ 若 PRD 有 `## 冲突待决` 节, 阻塞 `/plan`
- ☐ PRD 与产品文档对应章节做关键字段对比 (跳转路径 / 接口 / 字段名), 不一致必须显式标注

---

## 违规处理

### 检测时机

| 阶段 | 谁执行 | 检测内容 |
|-----|-------|---------|
| `/prd` 写 PRD | AI 自查 | R1-R5 全部 |
| `/prd-check` | 命令逻辑 | A / B / C 全部 |
| `/plan` 拆任务前 | AI 自查 | A (引用真实性), 若失败拒绝拆 |
| `/review` | code-reviewer | A / B / C 全部, 命中标 🔴 Critical |

### 修复优先级

R1-R3 违反: 🔴 Critical, 必须修才能继续下游
R4-R5 违反: 🟡 Warning, 应修但不阻塞

---

## 给 AI 自己的提示词

每次写下游产出 (PRD / 任务 / 代码) 前自问:

> **「这一行内容, 在上游文档里能找到原文吗?」**
> - 能 → 给出 file:line 引用
> - 不能 → **停下, 标「待评审」**, 不要自由发挥
> - 不能但「业界都这么做」 → 同上, **停下, 列两种方案, 让用户选**, 不要自行拍板

下游产出的**任何一句话**, 都应该可以反向追溯到上游某一行。**不允许「无源之水」**。
