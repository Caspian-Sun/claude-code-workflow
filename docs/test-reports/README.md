# test-reports 目录

> 存放由 `/test` 命令产出的**单次测试报告快照** — 一次运行一份文件, 不可变。

## 用途

每次执行 `/test` (生成新测试、重新生成过期测试, 或者只是跑回归), 都必须在这里落一份报告。这些报告闭合了**可追溯链**:

```
PRD → @rules → it() → 测试报告 (这里)
```

报告让「每条业务规则是否真的有绿色测试」**无需重跑测试**就可审计。对代码评审、周报交接、回顾, 以及跟踪规则覆盖率趋势都有用。

## 命名约定

```
docs/test-reports/<YYYY-MM-DD-HHmm>-<范围>.md
```

- `<YYYY-MM-DD-HHmm>` — 本轮**开始**运行的本地时间 (24 小时制)
- `<范围>` — kebab-case, 反映本次测试的范围:
  - 单文件 → 基名 (如 `search-form`)
  - 模块/目录 → 模块名 (如 `features-list`)
  - 全项目 → `all` 或 `multi`

示例:
- `2026-05-19-1430-search-form.md`
- `2026-05-19-1635-features-list.md`
- `2026-05-20-0900-all.md`

## 不可变性

- **绝不修改**已有报告。每份报告是一次测试运行的冻结快照。
- 重跑得到不同结果 → 用新时间戳写**新文件**。
- 趋势从时间序文件列表中自然浮现。

这与 `docs/retrospectives/` 的理念一致: 只追加的历史比一份不断覆盖的「最新」更有信号。

## 文件结构

骨架使用 [_template.md](_template.md)。必含章节:

1. **元信息** — 日期、目标 (`$ARGUMENTS`)、执行者、vitest 版本、总耗时
2. **执行汇总** — 总用例 / 通过 / 失败 / 跳过, 以及失败用例详情
3. **业务规则追溯矩阵** — 每个源文件的 `@rules` ↔ `it()` ↔ 状态。含数值化的**规则覆盖率**行。
4. **本轮变更** — 与上一份报告对比的新增/修改/删除用例
5. **未解决问题与建议** — 失败按类别分诊、未覆盖的规则、后续待跟进事项

## 与其他目录的关系

| 目录 | 角色 | 可变性 |
|------|------|--------|
| `docs/prds/` | `@rules` (业务规则) 的源头 | 需求变化时可编辑 |
| `docs/tasks/` | 任务拆解, 被 `@task` 引用 | status 字段可更新, 其余稳定 |
| `docs/bug-reports/` | **人 / E2E AI** 在运行态发现的缺陷 | 只追加 |
| `docs/test-reports/` (本目录) | `/test` 单测/组件测试跑出的失败与覆盖快照 | 只追加 |
| `docs/retrospectives/` | `/meta-audit` 的框架级审计 | 只追加 |

**bug-reports vs test-reports**: bug-reports 流向 `/fix` (描述用户可见的、需修复的缺陷)。test-reports 流向代码评审 / 回顾 (描述自动化覆盖现状)。两者互不替代。
